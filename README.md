# 🚀 Локальный GitOps Конвейер: Terraform + Kubernetes + GitHub Actions

Проект реализует автоматическое развертывание и масштабирование инфраструктуры на локальном кластере Kubernetes с использованием методологии **GitOps** и принципа **IaC (Infrastructure as Code)**.

## 🛠 Технологический стек
* **Оркестрация:** Kubernetes (v1.32.0)
* **Управление пакетами:** Helm3 (v3.20.2) + Инфраструктурный чарт `Ingress-Nginx`
* **Автоматизация инфраструктуры:** Terraform (с бэкендом хранения состояния в K8s)
* **CI/CD Автоматизация:** GitHub Actions + Локальный фоновый Self-Hosted Runner
* **Безопасность реестров:** Локальное зеркало провайдеров Yandex Cloud (для обхода блокировок HashiCorp)

---

## 🏗 Архитектура решения

1. **Код как источник правды:** Вся конфигурация описана в файле `main.tf`.
2. **Безопасный State:** Файл состояния Terraform (`terraform.tfstate`) не хранится в Git и не привязан к локальному ПК. Он автоматически сохраняется в виде защищенного Secret внутри системного пространства имен `kube-system` вашего Kubernetes-кластера.
3. **Локальный чарт:** Во избежание сетевых блокировок публичных реестров, Helm-чарт `ingress-nginx` скачан локально в репозиторий.
4. **Контур автоматизации:** При пуше в ветку `main`, GitHub передает задачу локальной службе раннера, которая раскатывает изменения прямо на вашем ПК.

---

## 🚀 Как запустить локально

Если вы хотите развернуть или проверить конфигурацию вручную без Git:

```bash
# 1. Инициализация и подключение к бэкенду в K8s
terraform init -reconfigure

# 2. Проверка планируемых изменений
terraform plan

# 3. Применение конфигурации (развертывание 2-х реплик Nginx)
terraform apply -auto-approve
```

## 🤖 CI/CD Автоматизация (GitHub Actions)

Пайплайн развернут на уровне **Организации GitHub** и поддерживает два режима работы:

1. **Авто-деплой (Push в main):** Автоматически накатывает изменения и масштабирует поды.
2. **Ручное управление (Workflow Dispatch):** На вкладке *Actions* доступна кнопка запуска с возможностью выбора действия:
   * `apply` — плановое развертывание/обновление.
   * `destroy` — полная и чистая очистка кластера от созданных ресурсов.

## 🔍 Проверка статуса инфраструктуры

После успешного выполнения пайплайна, проверьте состояние ресурсов в кластере:

```bash
# Проверка запущенных подов в пространстве имен production
kubectl get pods -n production

# Проверка выделенного IP и портов балансировщика
kubectl get svc -n production
```
